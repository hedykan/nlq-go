# LLM响应速度影响因素分析

## 📊 当前性能情况

根据你的项目配置和代码分析，以下是影响LLM响应速度的关键因素：

---

## ⚡ 直接影响响应速度的因素

### 1. **模型选择** - 影响最大！⭐⭐⭐⭐⭐

**当前配置：**
```yaml
model: glm-4.7  # 慢速模型
```

**问题分析：**
- `glm-4.7` 是推理能力强但响应较慢的模型
- 你的注释提到`glm-4-flash`是闪电模型，快20倍！

**性能对比：**
- `glm-4-flash`: ~1秒响应 ⚡
- `glm-4.7`: ~20秒响应 🐌

**建议：**
- 对于表选择（阶段1），使用`glm-4-flash`
- 对于SQL生成（阶段2），使用`glm-4.7`

**预期提升：** 阶段1速度提升**95%**（14.5秒 → ~1秒）

---

### 2. **MaxTokens（最大生成token数）** - 影响中等 ⭐⭐⭐

**当前配置：**
```go
Temperature: 0.1,
MaxTokens:   1000,  // GenerateSQL中
MaxTokens:   4096,  // GenerateContent中
```

**问题分析：**
- `MaxTokens`设置越大，LLM生成的token越多，响应越慢
- 1000对于简单SQL生成可能过多
- 4096对于表选择（返回JSON）可能过多

**建议：**
- 表选择（阶段1）：`MaxTokens: 500`（JSON响应通常很短）
- SQL生成（阶段2）：`MaxTokens: 1000`（保持不变）

**预期提升：** 减少10-20%的响应时间

---

### 3. **重试机制** - 失败时影响大 ⭐⭐⭐⭐

**当前配置：**
```go
maxRetries: 3,  // 最多重试3次
```

**问题分析：**
- 每次失败后会重试，最多3次
- 如果前2次都失败，第3次才成功，总耗时 = 单次耗时 × 3
- 指数退避策略：第1次重试等2秒，第2次等4秒

**你的日志显示：**
- 没有重试发生（说明API调用稳定）
- 这个不是当前性能瓶颈

**建议：**
- 保持当前设置（稳定性优先）
- 如果追求极致性能，可以考虑降低到2次

---

### 4. **超时设置** - 间接影响 ⭐⭐

**当前配置：**
```yaml
llm:
  timeout: 90s  # LLM调用超时

server:
  read_timeout: 120s
  write_timeout: 120s
```

**HTTP客户端配置：**
```go
httpClient: &http.Client{Timeout: 90 * time.Second}
```

**问题分析：**
- 超时设置过长不会让响应变快
- 但设置太短可能导致频繁超时重试
- 90秒是合理的（考虑到glm-4.7的慢速特性）

**建议：**
- 如果切换到`glm-4-flash`，可以缩短到30秒
- 保持当前设置（稳定性优先）

---

### 5. **Temperature（温度参数）** - 影响较小 ⭐

**当前配置：**
```go
Temperature: 0.1
```

**问题分析：**
- Temperature控制输出的随机性
- 0.1表示确定性输出（低随机性）
- 较低的temperature通常响应更快（因为不需要更多采样）

**建议：**
- 保持当前设置（SQL生成需要确定性）
- 不影响性能

---

### 6. **两步法本身** - 架构层面的影响 ⭐⭐⭐⭐⭐

**当前实现：**

- 阶段2：SQL生成（~24.9秒）
- **总计：~40秒（2次LLM调用）**

**单步法对比：**

**性能差距：60%**

**建议：**
- ✅ 已实现智能切换（feat/v0.3）
- 小型数据库使用单步法
- 大型数据库使用两步法
